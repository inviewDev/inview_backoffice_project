# I&VIEW BackOffice 리뉴얼 작업 종합 보고서

- 작성 기준일: 2026년 6월 24일
- 대상 프로젝트: I&VIEW BackOffice
- 기술 구성: React, React Router, Express, Prisma, PostgreSQL
- 작업 상태: 주요 화면·업무 기능·기존 광고 데이터 이전 완료, 운영 QA 진행 중

## 1. 작업 개요

기존 백오피스의 광고 등록·관리 데이터를 신규 React 기반 관리자 시스템으로 이전하고, 관리자 업무 흐름에 필요한 회원 관리, 광고 등록, 광고 관리, SMS 계약 동의, 마이페이지, 대시보드 기능을 정비했다.

화면은 Figma 기획안을 기준으로 재구성했으며, 최근 작업에서는 Dashboard, 광고관리, 광고관리 상세, 광고등록 화면을 `width: 100%`, `max-width` 기반의 Flex 레이아웃으로 변경해 반응형 대응 범위를 확대했다.

## 2. 주요 기능 작업 내역

### 2.1 공통 구조 및 인증

- React Router 기반 관리자·계약서 페이지 라우팅 정리
- 공통 사이드바와 내비게이션 렌더링 구조 정비
- 사이드바 접기·펼치기 및 상태 로컬 저장 적용
- Font Awesome 단일 Chevron 아이콘 적용
- JWT 사용자 식별값, 권한, 만료시간 처리
- 비밀번호 찾기·재설정 요청 및 토큰 처리 API 구성
- 사용자별 접근 권한과 관리자 조회 범위 분기

- 생년월일 선택용 공통 DatePicker 레이어 디자인 적용
- 마이페이지 생년월일 수정 기능 연동

- 개인 일정 등록·삭제 및 달력 표시
- 일정 시작·종료 DatePicker 레이어 디자인 통일

### 2.3 광고등록

- 기본정보등록에서 불필요한 생년월일 항목 제거
- TEL 첫 구간을 4자리까지 입력 가능하도록 확장해 `0507` 계열 번호 지원
- 전화번호, 사업자번호, 카드번호 구간 입력 완료 시 다음 칸 자동 포커스 이동
- 계약기간 DatePicker에 연도·월 선택 헤더 및 공통 레이어 디자인 적용
- 카드사 선택 목록 추가
- 카드 할부 개월 수를 1개월부터 12개월까지 확장
- 담당자를 로그인 사용자 이름으로 기본 고정
- 담당팀장 및 담당부장·대표·파트장 선택 목록 구성
- 상품정보를 상품 1~10까지 저장할 수 있도록 확장
- 상품정보 표시 순서를 좌우 2열 기준으로 정리
- 세금계산서, 결제상태, 제작사항, 광고진행, 광고주계정 등 상세 입력 지원
- 계약 등록 시 VAT와 순이익 계산 및 결제·상품·추가정보 저장

### 2.4 광고관리 및 상세

- TanStack React Table 기반 정렬, 필터, 페이지네이션 적용
- 기본 페이지당 조회 건수를 10건으로 변경

- 컬럼 헤더 클릭 방식 오름차순·내림차순 정렬
- 행 클릭으로 광고 상세 페이지 이동
- 상세 페이지에서 기본정보, 결제정보, 상품정보 1~10, 추가정보 표시
- 광고등록 화면과 광고관리 상세 화면의 시각적 구조 통일

### 2.5 SMS 계약서 및 동의

- 광고 상세에서 SMS 계약서 최초 발송과 재발송 지원
- 재발송 시 사용 완료 토큰 또는 유효 토큰 재사용

- 동의 완료된 계약서는 만료 여부와 관계없이 재조회 가능
- 이미 동의된 링크 재접속 시 계약 내용을 다시 확인 가능
- 동의 완료 후 회사 홈페이지로 이동하도록 처리
- 동의일시, 동의 IP, 동의상태를 결제 데이터와 토큰에 기록
- 문자 계약서에 상품정보 1~10을 포함하도록 구성

### 2.6 Dashboard

- 현재 연도 기준 최근 3개 연도 매출 데이터 제공
- 기본 활성 연도를 현재 연도로 설정

- 현재 연도 파란색, 전년도 빨간색, 전전년도 초록색 적용
- 월별 그래프 포인트 호버 툴팁 안정화
- 매출취소·결제대기 상태를 제외한 완료 매출 집계
- 오늘 실적 Top 10 및 이번 달 실적 Top 10 제공
- 담당자별 매출액 기준 막대 차트와 툴팁 적용
- 공지사항, 일정, 실적 패널 높이 및 제목 배치 정리
- 매출현황 검색, 기간 단축 버튼, 합계, 상태별 행 색상, 페이지 푸터 구성

### 2.7 반응형 레이아웃

## 3. 기존 광고 데이터 이전 방식

### 3.1 이전 방식 요약

업무 데이터는 FTP 디렉터리의 파일을 그대로 복사한 방식이 아니다. 기존 백오피스가 반환하는 광고 목록 JSON과 광고 상세 조회 응답을 추출한 뒤, 신규 DB 구조에 맞게 정제·매핑·적재하는 ETL 방식으로 진행했다.

FTP 접속정보는 기존 시스템 확인 과정에 사용됐으며, 계정과 비밀번호는 소스 코드 및 본 보고서에 저장하지 않았다.

### 3.2 1차 목록 데이터 추출

- 원본 파일: `.tmp_migration/inviewcc_ads_export.json`
- 원본 광고 건수: 14,027건
- 고유 광고코드 수: 14,027개
- 목록 원본 필드 수: 28개
- 주요 필드: 담당자, 부서, 상호명, 대표자, 사업자번호, 전화, 휴대폰, SMS 상태, 동의상태·일시, 계약기간, 상품, 결제금액, VAT, 소진비, 순이익, 결제구분, 카드사, 결제상태, 제작사항, 광고진행, 광고주계정, 승인번호, 작성일

원본 JSON 내부의 탭·개행 등 제어문자로 인한 파싱 오류를 방지하기 위해 `sanitizeJsonText`로 문자열 내부 제어문자를 이스케이프한 후 파싱했다.

### 3.3 1차 신규 DB 적재

실행 스크립트: `scripts/import-legacy-ads.js`

1. 목록 JSON을 읽고 공백, 금액, 날짜 형식을 정규화한다.
2. 담당자명과 부서 조합을 기준으로 기존 사용자를 찾는다.
3. 일치 사용자가 없으면 로그인이 불가능한 임의 비밀번호의 레거시 사용자 계정을 생성한다.
4. 회사 정보는 `Company`, 광고·결제 정보는 `Payment`로 분리한다.
5. 회사와 결제 데이터는 광고 건별 Prisma 트랜잭션으로 함께 생성한다.
6. 500건 단위로 진행 상황을 출력한다.

담당자·부서 조합은 총 242개이며 현재 DB에 242개의 레거시 사용자 계정이 연결돼 있다.

### 3.4 중복 방지 방식

다음 값을 조합해 중복 판정 키를 생성했다.

`사업자번호 + 상호명 + 작성일 + 결제금액 + 상품명 + 승인번호`

적재 전 기존 `Payment` 데이터를 조회해 동일 키가 있으면 생성 대상에서 제외한다. 2026년 6월 24일 드라이런 결과 원본 14,027건 전부가 기존 DB 데이터와 중복 판정돼 추가 생성 대상은 0건이었다.

### 3.5 2차 상세정보 수집

실행 스크립트: `scripts/import-legacy-ad-details.js`

- 기존 상세 조회 주소에 광고코드를 POST 방식으로 전달
- 조회 식별자: `Select_AdvertisingDetail`
- 기본 동시 요청 수: 6개
- 요청 시간 제한: 20초
- 실패 시 최대 3회 재시도
- 전체 응답을 `.tmp_migration/inviewcc_ad_details_export.json`에 캐시

상세 데이터는 총 14,027건이며 조회 실패는 0건이다. 상세 원본은 60개 필드로 구성돼 있다.

### 3.6 상세 데이터 보강

상세 응답을 광고코드와 1차 중복 판정 키로 신규 결제 데이터에 연결한 뒤 다음 정보를 갱신했다.

- 회사: 상호명, 대표자, 사업자번호, 전화, 휴대폰, 우편번호, 주소, 상세주소, 홈페이지, 이메일
- 결제: 계약 시작·종료일, 결제금액, VAT, 소진비, 순이익, 승인회사, 세금계산서, 카드 할부, 승인번호
- 담당: 담당자, 담당팀장, 담당부장·대표·파트장
- 상품: 상품 1~10을 `Payment.productItems` JSON 배열로 저장
- 추가정보: 광고주계정, 등록 URL, 제목 문구, 설명 문구, 메모

회사와 결제 상세 갱신도 건별 Prisma 트랜잭션으로 처리해 일부 필드만 반영되는 상태를 방지했다.

- `Payment`에 결제금액, VAT, 소진비, 순이익, 승인번호, 결제상태, 카드사, 할부개월, 담당자 계층, 제작사항, 광고진행, 광고주계정, 추가 문구 필드 추가

- 비밀번호 재설정 토큰 테이블 추가
- 개인 메모, 일정, 급여, 매출·취소 상세 모델 추가

## 5. 검증 및 QA

- 데이터 이전 스크립트는 기본적으로 Dry-run이며 `--apply` 지정 시에만 DB를 변경한다.
- 목록 적재 전 중복 키와 생성 예정 건수를 출력한다.
- 상세 갱신 전 매칭, 미매칭, 누락, 중복 후보 및 주요 필드 보유 건수를 출력한다.
- 목록 14,027건과 상세 14,027건의 재검증 Dry-run 완료
- Vite 프로덕션 빌드 완료
- Dashboard, 광고관리, 광고관리 상세, 광고등록 CSS에서 Grid 선언 제거 확인

## 6. 운영 전 권고사항

1. 중복 후보로 분류된 광고 3건은 기존 광고코드와 신규 Payment ID를 수동 대조한다.
2. SMS 운영 발송 전 발신번호, 운영 도메인, 회사 홈페이지 환경변수를 재확인한다.
3. 마이그레이션 완료 후 전달된 FTP 및 기존 백오피스 비밀번호를 변경한다.
4. `.tmp_migration` 원본 파일은 개인정보가 포함돼 있으므로 저장소에 커밋하지 않고 접근 권한이 제한된 백업 영역으로 이동한다.
5. 운영 배포 전 데스크톱·태블릿·모바일 해상도별 시각 QA와 주요 업무 시나리오 회귀 테스트를 진행한다.
